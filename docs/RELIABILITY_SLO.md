# Reliability SLO and 24/7 Operations Playbook

## Objective

Operate the platform with continuous monitoring, bounded failure impact, and deterministic recovery behavior.

This is an operational reliability document, not a legal or compliance policy.

## Service Level Objectives

| SLI | Target | Window |
|---|---:|---|
| API availability (critical health endpoints) | 99.9% | rolling 30 days |
| p95 latency (critical health endpoints) | <= 1000 ms | rolling 24 hours |
| Synthetic monitor pass rate per run | 100% critical checks | each run |

## Error Budget

At 99.9% monthly availability, the error budget is approximately 43.2 minutes per 30-day month.

Budget policy:
1. If remaining budget drops below 50%, pause non-critical production changes.
2. If remaining budget drops below 25%, freeze feature rollouts and focus only on reliability fixes.
3. If budget is exhausted, enforce incident-only changes until stability is restored.

## Monitoring Sources

1. Public and dependency health endpoints.
2. Synthetic monitor artifacts generated by script and CI workflow.
3. Slack and Datadog alerts from backend monitoring service.
4. Self-heal status endpoint for autonomous recovery state visibility.

## Self-Heal Behavior

Self-heal loop runs every 5 minutes through Celery Beat and performs safe actions only:
1. Database probe + optional table verification.
2. Redis and queue pressure checks when Redis/Celery broker is configured.
3. Cache warming under queue pressure.
4. Optional autonomy freeze on sustained infra failure.

Config flags:
1. SELF_HEAL_ENABLED (default true)
2. SELF_HEAL_FAILURE_THRESHOLD (default 2)
3. SELF_HEAL_QUEUE_DEPTH_THRESHOLD (default 3000)
4. SELF_HEAL_AUTO_FREEZE_ON_FAILURE (default true)

## Incident Severity Model

| Severity | Definition | Target Response |
|---|---|---|
| SEV-1 | Critical checks down, major user impact | immediate page, active mitigation |
| SEV-2 | Partial degradation, core path still works | triage within 15 minutes |
| SEV-3 | Non-critical failures or latency drift | fix in planned reliability cycle |

## Auto-Remediation Playbooks

### Playbook A: Critical Health Failure

1. Check synthetic report and identify failing critical endpoint.
2. Check API process logs and deployment events.
3. Confirm database connectivity and credentials.
4. Confirm self-heal escalation state.
5. If sustained failures continue, keep autonomy frozen and route to manual operations.

### Playbook B: Queue Pressure and Latency Drift

1. Check queue depth and worker availability.
2. Confirm cache warm cycles are succeeding.
3. Inspect top slow endpoint classes and DB latency.
4. Scale worker or API replicas as needed.
5. Close incident only after synthetic checks return to full pass.

### Playbook C: Recovery Validation

1. Confirm synthetic monitor status is pass.
2. Confirm self-heal consecutive_failures has reset.
3. Confirm alert stream is quiet for one full monitor interval.
4. Record timeline and root cause in incident notes.
